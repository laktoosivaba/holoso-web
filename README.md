# holoso-web

Browser UI for [holoso-synth](../holoso-synth): write a numeric Python kernel, get synthesizable Verilog — with the
**synthesis running entirely in your browser** via [Pyodide](https://pyodide.org) (CPython compiled to WASM).

## Layout

| path | role |
|---|---|
| `index.html` | UI shell (editor, example/entry pickers, controls, output tabs, log) |
| `app.js` | UI controller; talks to the worker over `postMessage`. Never touches Python. |
| `worker.js` | Web Worker: boots the vendored Pyodide, installs the wheel, runs synthesis off the UI thread |
| `driver.py` | the synthesis logic (`synth_to_json`, `demos_to_json`), exec'd inside Pyodide. **Single source** shared by the worker and the Node test in `tools/`. |
| `yosys-worker.js` · `yosys-job.js` | lazy Web Worker for the in-browser FPGA flow: Yosys resource **Estimate** (gate/cell histogram) and ECP5 **Route** (nextpnr place&route → real Fmax + post-route utilization). Pure helpers in `yosys-job.js` are shared with the Node tests. |
| `closure.js` | computes the RTL instantiation closure (which Kulibin sources a design needs) — shared by the worker and tests |
| `tools/` | Node-Pyodide tooling: driver regression test (`driver_test.mjs`), Pyodide vendoring (`vendor-pyodide.mjs`) + its check (`vendor-check.mjs`); plus the Yosys/nextpnr worker tests + spikes |
| `Makefile` | build orchestration (`dist`, `wheel`, `vendor`, `test`, `serve`) |
| `Dockerfile` · `deploy/` | container image (nginx serving `dist/`) + nginx config |
| `demos/` | demo kernels for the example picker — static `.py` sources + `manifest.json` |
| `wheels/` · `pyodide/` · `yosys/` · `nextpnr-ecp5/` · `hdl/` · `dist/` | build artifacts — git-ignored, regenerated by `make dist` |

The FPGA flow runs each tool as its own WASM module in the worker, fetched lazily so neither is on the page's
critical path: **Estimate** pulls ~50 MB of Yosys on first use, **Route** ~170 MB of ECP5 chipdb. Route works on
the raw kernel, which holoso exposes with every operand/result as a top-level port — so it is I/O-pad-bound, and a
design wider than the package's pads surfaces a place-and-route error rather than a number.

**Demo kernels** live in `demos/` as plain `.py` source files (`dot2`, `madd`, `cube`, `poly3`, `blend`,
`ekf_update`), listed by `demos/manifest.json`. The worker fetches them to populate the picker — no engine
round-trip and no dependency on the holoso wheel. Add a kernel by dropping a `.py` file in `demos/` and adding a
manifest entry.

## Test (headless)

```bash
make test    # driver regression (every demo synthesizes; error paths; entry selection)
             # + vendored-runtime check (boots Pyodide offline from pyodide/ and runs the driver)
```

## Build

`make dist` assembles a self-contained static site into `dist/` (static files + the holoso wheel + the vendored
Pyodide, Yosys, and nextpnr-ecp5 runtimes) — host it with any plain file server.
